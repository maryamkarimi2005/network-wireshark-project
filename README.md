### شرح مراحل انجام‌شده

1. Wireshark نصب و اینترفیس **Wi-Fi** به‌عنوان اینترفیس فعال شبکه انتخاب و فرآیند Capture آغاز شد.

2. در ترمینال دستور زیر برای ارسال یک درخواست HTTP خام (بدون رمزنگاری TLS) اجرا شد:
```
curl http://example.com
```
> **نکته:** ابتدا از `neverssl.com` استفاده شد، اما به دلیل محدودیت دسترسی شبکه (بدون VPN)، درخواست با خطای Timeout مواجه شد. به همین دلیل، از دامنه‌ی جایگزین `example.com` استفاده شد که همانند `neverssl.com` بر پایه‌ی HTTP خام (بدون ریدایرکت به HTTPS) پاسخ می‌دهد.

3. پس از دریافت پاسخ کامل HTML در ترمینال، فرآیند Capture در Wireshark متوقف شد.

4. با فیلتر `ip.addr == 172.66.147.243`، چرخه‌ی کامل ارتباط شامل هندشیک سه‌طرفه TCP، درخواست HTTP، پاسخ سرور و بستن ارتباط (FIN/ACK) بررسی و صحت کپچر تأیید شد.

## فاز سوم: تحلیل رفتار سرور و زمان‌بندی RTT

### پکت مورد بررسی
پکت پاسخ سرور به درخواست GET: **پکت شماره ۱۲۴**

### استخراج و تحلیل فیلدها

| فیلد | مقدار | توضیح |
|---|---|---|
| Status Code | `200 OK` | نشان‌دهنده‌ی موفقیت کامل درخواست است؛ سرور منبع درخواستی را بدون خطا پیدا و بازگردانده است (رده‌ی موفقیت 2xx) |
| Time Delta (RTT) | `137.642900 ms` | فاصله‌ی زمانی بین ارسال درخواست GET (پکت ۱۲۱) و دریافت پاسخ (پکت ۱۲۴) |
| Server | `cloudflare` | سرور پشت شبکه‌ی توزیع محتوای Cloudflare قرار دارد که تأخیر پایین این پاسخ را توجیه می‌کند |

![Status Code and RTT](screenshots/phase3_status_code_rtt.png)

*شکل ۵: جزئیات فریم پاسخ (پکت ۱۲۴) شامل Status Code و Time Delta (RTT)؛ فیلدهای کلیدی با کادر رنگی مشخص شده‌اند.*

### سناریوی تحلیلی: کندی RTT بیش از ۲ ثانیه

در این آزمایش، مقدار RTT برابر با **۱۳۷.۶۴ میلی‌ثانیه** بوده که نشان‌دهنده‌ی یک اتصال سالم و پرسرعت است. اما در سناریویی فرضی که این مقدار بیش از ۲ ثانیه طول بکشد، ریشه‌ی مشکل می‌تواند در چند لایه باشد. اگر تأخیر ثابت و بدون نوسان زیاد در تمام درخواست‌ها تکرار شود، احتمالاً مشکل از **مسیر شبکه یا پهنای باند اتصال کلاینت** است (مثلاً استفاده از VPN با مسیر طولانی، یا شبکه‌ی موبایل ضعیف). اگر تأخیر فقط در برخی درخواست‌ها و نه به‌صورت پیوسته دیده شود، احتمالاً علت به **پردازش داخلی سرور** بازمی‌گردد؛ برای مثال درخواست به دیتابیس، تولید محتوای پویا، یا Rate Limiting سمت سرور. همچنین بار زیاد هم‌زمان روی سرور (مثل ترافیک بالا در ساعات پیک) می‌تواند باعث صف‌بندی درخواست‌ها و افزایش Time Delta شود. برای عیب‌یابی دقیق، می‌توان با ابزارهایی مانند `curl -w "%{time_total}"` یا اجرای چندباره‌ی درخواست و مقایسه‌ی زمان‌ها، تشخیص داد که کندی مربوط به شبکه است یا سمت سرور.
